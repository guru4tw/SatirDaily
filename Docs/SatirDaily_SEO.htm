SatirDaily — SEO 優化彙整

給其他 AI / 顧問檢視，徵求進一步可行的改善方向。

網站 guru4tw.github.io/SatirDaily 原始碼 github.com/guru4tw/SatirDaily 類型 全靜態 GitHub Pages（前端 SPA） 更新日 2026-06-23

摘要（TL;DR）

SatirDaily 是「全台薩提爾活動／課程／工作坊每日自動匯整清單」，全靜態託管於 GitHub Pages，由 GitHub Actions 每日爬取各機構公開活動、寫入 events.json，前端純瀏覽器端排序／搜尋／篩選。使用者意圖屬交易型（找工作坊、要報名）。

基礎與技術型 SEO 已大致到位：已解決 SPA「原始 HTML 空白」這個最致命的問題（活動清單與結構化資料皆伺服器端預渲染進原始 HTML）、robots/sitemap/GSC 驗證／Event JSON-LD／意圖化 metadata 皆完成。目前瓶頸轉為「站外權重」與「單頁 SPA 的長尾覆蓋上限」，這正是想徵求其他 AI 進一步建議之處（見 §5）。

1網站定位與目標

項目內容
產品本質薩提爾相關活動／課程匯整目錄（非金句站、非部落格）。每日主動爬取，依日期遠近排序。
目標受眾對薩提爾模式有興趣、想報名工作坊／課程的台灣使用者（家長、伴侶、助人工作者、自我成長者）。
搜尋意圖交易型 / 導航型：薩提爾工作坊、薩提爾課程、〔講師〕工作坊、〔地區〕薩提爾課程、薩提爾 報名。
商業模式非營利資訊匯整，零成本（全免費靜態架構）。目標是自然流量 + 被薩提爾社群信任引用。
內容宗旨資料正確、可溯源（每筆連回原始活動頁並驗證連結）；正面回饋畫龍點睛，不弱化成冷條列。

2技術架構（與 SEO 的關係）

層角色SEO 影響
GitHub Actions每日 cron（台北 05:00）爬蟲 → 更新 JSON → 自動 commit內容每日更新（Google 偏好「活著」的站）
Python 爬蟲 + build抓 5 來源活動 → events.json；build_site_json.py 正規化並把清單＋JSON-LD 預渲染進 index.html原始 HTML 含可索引內容＋結構化資料
GitHub Pages純靜態 HTML/JS，瀏覽器端排序／搜尋／篩選載入快、自帶 CDN/HTTPS；但本質是單一 URL 的 SPA

關鍵限制：單頁 SPA

所有活動都渲染在同一個首頁 URL，沒有「每場活動／每位講師獨立網址」。預渲染讓 Google 看得到內容，但長尾關鍵字（特定講師、特定主題、特定地區）缺乏專屬落地頁可排名 —— 這是目前架構的 SEO 上限之一（見 §5 待解）。

4現況數據與已知弱點

現況數據

  • 活動筆數：77（lopwilldo 26、satir_org 28、shiuhli 18、comflow 3、omia 2）；其中 72 筆有固定日期，已進 Event JSON-LD。
  • 更新頻率：每日 1 次（台北 05:00，GitHub Actions 自動）。
  • 連結品質：每筆 build 時 verify_links.py 驗證原始活動頁有效性。
  • 收錄狀態：GSC 已驗證；sitemap 尚待在 GSC 提交（這是目前最即時待辦）。

已知弱點（徵求改善的重點）

弱點說明 / 影響
單頁 SPA、單一 URL所有活動同一頁，無法為「特定講師／主題／地區」提供專屬落地頁，長尾關鍵字覆蓋受限。
子網域、無自訂網域guru4tw.github.io/SatirDaily 品牌性與信任度低於獨立網域，且與 github.io 共享網域權重。
新站、零反向連結網域權重低，無外部引用，初期排名難起。
無 og:image / 圖片素材社群分享無縮圖，CTR 受限；圖片搜尋無曝光。
內容深度仍偏薄除活動清單外，主題知識內容少（僅一段介紹＋2 FAQ），主題權威性不足。
過期活動混入清單含已過去活動，可能稀釋「即將舉行」的相關性訊號。
JSON-LD 為 ItemList 非獨立 Event 頁Google 對 Event rich result 偏好「一活動一頁」，ItemList 取得完整活動卡片結果的機率較低。

5給其他 AI 的諮詢提示（可整段複製）

把以下整段貼給另一個 AI（連同本檔前述脈絡），請它在「維持全免費、純靜態 GitHub Pages、無後端」的硬限制下，提出可落地的進一步 SEO 改善。

我有一個全靜態網站 SatirDaily（https://guru4tw.github.io/SatirDaily/），是「全台薩提爾活動／課程每日自動匯整清單」，託管於 GitHub Pages，由 GitHub Actions 每日爬取活動寫入 events.json，前端是單頁 SPA。使用者意圖是交易型（找薩提爾工作坊、要報名）。 【已完成的 SEO】意圖化 title/description/keywords、canonical、Open Graph、robots.txt、自動產生的 sitemap.xml、Google Search Console 擁有權驗證、伺服器端預渲染 77 場活動進原始 HTML、Event JSON-LD（ItemList，72 筆）、h1 sr-only 描述、關於薩提爾長文＋FAQ、LINE/FB 分享鈕、每日自動把更新寫進 index.html。 【硬限制】必須維持：全免費、純靜態（GitHub Pages，無伺服器/資料庫/付費服務）、繁體中文台灣市場、薩提爾利基。可改前端 HTML/JS 與 Python build 腳本（Actions 每日跑）。 【已知弱點】1) 單頁 SPA、單一 URL，長尾關鍵字無專屬落地頁；2) 子網域無自訂網域；3) 新站零反向連結；4) 無 og:image；5) 內容深度偏薄；6) 過期活動混入；7) JSON-LD 是 ItemList 而非一活動一頁。 【請提供】針對上述，列出「投入產出比最高」的進一步 SEO 改善清單，每項請含：(a) 預期效益與原理、(b) 在純靜態/GitHub Pages 下的具體實作方式（可用 build 腳本生成）、(c) 風險或維護成本、(d) 優先級。特別想知道：是否值得在 build 時為每場活動／每位講師／每個地區「靜態生成獨立 HTML 頁」以擴張長尾與 Event rich result？若值得，請給出資料夾結構、URL 規劃、sitemap 與內部連結策略；若不值得，請說明替代方案。

提示：不同 AI 會給出不同切角，建議至少問 2～3 家並交叉比對；落地前再回到本專案做可行性與成本評估。
