Project Plan · 計畫書

全台薩提爾活動，
每天替你自動匯整一次

一個會自己更新的活動清單網站。每天主動搜尋全台薩提爾相關活動，依日期遠近排序，並提供自訂排序與多關鍵字搜尋，讓人輕鬆找到屬於自己的那一場。

託管方式GitHub Pages
更新方式GitHub Actions 每日排程
資料格式events.json
總成本NT$ 0
01 — Architecture

整體架構

關鍵觀念：GitHub Pages 只負責「顯示」，不會自己跑程式。「每天主動更新」由 GitHub Actions 這個內建免費排程器完成——它定時跑爬蟲、更新 JSON、自動提交，網站隨之刷新。三者全部免費。

Scheduler
GitHub Actions 每日定時觸發（cron），執行爬蟲腳本
→
Crawler
Python 爬蟲 抓各來源活動，正規化後寫入 events.json 並 commit
→
Frontend
GitHub Pages 讀取 JSON，前端負責排序、搜尋、呈現

前端是純靜態的 HTML + JavaScript，排序與篩選都在瀏覽器端完成。唯一例外是「登入與愛心評價」這類需要跨使用者共享、持久保存的互動資料——它無法寫進每日被覆寫的 events.json，改由 Firebase 這個免費託管後端承接（見 §08）。前端仍不需自己架伺服器。

02 — Data Sources

資料從哪裡來

薩提爾活動沒有單一官方來源，分散在各協會、帶領者與售票平台。策略是「主來源優先 + 逐步擴充」，每個來源各寫一支解析器。

建議優先抓取的來源

  • 售票/報名平台：Accupass、KKTIX——結構化、有 API 或穩定的 HTML，最好抓，建議第一階段先做。
  • 協會官網：如台灣薩提爾人文發展中心、各地家族治療／薩提爾推廣協會的課程頁。
  • 知名帶領者：個別講師的官網或公開課程頁。
  • Facebook 公開活動：涵蓋最廣，但反爬蟲嚴格、條款限制多，列為後期、低優先，或改用人工補登。

關鍵字策略

以「薩提爾、Satir、冰山、家族治療、自我覺察、對話練習、一致性表達」等關鍵字交叉搜尋各平台，再過濾雜訊。

04 — Features

排序與搜尋功能

預設依日期遠近排序（最近的活動在最上方），並開放自訂排序與多維搜尋。全部在前端即時運算。

預設排序

依 date_start 由近到遠，過期活動自動沉底或隱藏。

自訂排序

可切換依日期、地區、帶領者、主辦協會、價格排序。

關鍵字搜尋

跨標題、宗旨、內容、帶領者、協會的即時全文搜尋。

多重篩選

依地區、價格區間、月份多條件交叉過濾。

正評優先呈現

卡片著重顯示宗旨與參加者感謝，弱化冷冰冰的條列資訊。

RWD 響應式

手機、平板、桌機皆適配，無障礙鍵盤可操作。

05 — Daily Pipeline

每日更新流程

為了解決「資訊過期」，每天主動跑一次完整流程。第一步先重建來源清單，再依清單逐一抓活動，最後刷新網站。

Step 1 · 來源清單
discover_sources.py 讀人工維護的 Satir_website.md（純網站連結），逐站做連線檢查與結構偵測，產出 parser 友善的 Satir_source.md（含結構線索與建議解析路徑）。不抓活動內容。
→
Step 2 · 逐站爬取
各網站專屬爬蟲 讀 Satir_source.md，依各來源的「建議解析路徑」跑對應 parser（如 crawlers/shiuhli.py），正規化後寫入 events.json。
→
Step 3 · 發佈
commit + Pages GitHub Actions 自動提交更新後的 JSON，GitHub Pages 隨之刷新，前端依日期排序呈現。

為什麼把「來源清單」和「活動爬取」分開？

人工只維護純連結清單 Satir_website.md（變動慢、好讀好改）；腳本把它轉成含結構線索的 Satir_source.md（給機器與 parser 用）；活動資料變動最快，由各爬蟲抓。三層分開後，新增一個機構只要在 Satir_website.md 加一行，再替它寫一支獨立爬蟲即可，互不干擾，也方便追蹤哪些來源已可爬、哪些待開發。旭立文教基金會已勘查完成（課程頁含「薩提爾」分類標籤、日期、價格、講師與分頁），列為第一個待開發爬蟲。

06 — Build Steps

開發步驟

  1. 建立 repo 與 GitHub Pages放入前端骨架與一份手動填好的 events.json 範例，先讓網站能顯示。
  2. 來源清單腳本 discover_sources.py讀人工維護的 Satir_website.md，逐站連線檢查 + 結構偵測，產出 parser 友善的 Satir_source.md。已可運作，旭立列為第一個待開發來源。
  3. 完成前端排序與搜尋純 JS 讀取 JSON，實作預設排序、自訂排序、搜尋、篩選與卡片版型。
  4. 寫第一支網站爬蟲 crawlers/shiuhli.py解析旭立課程頁（薩提爾分類、日期、價格、講師、分頁），輸出符合資料模型的 JSON。
  5. 設定 GitHub Actions 每日排程cron 每日觸發：先跑 discover_sources.py 重建清單，再依清單跑各爬蟲，去重後自動 commit。
  6. 逐步擴充來源與爬蟲在 SEED_SOURCES 新增機構並各寫一支 parser；加入人工補登（編輯 JSON / issue 表單）。
  7. 接 Firebase 登入與愛心建 Firebase 專案，開 Google 登入、設授權網域與 Firestore 安全規則；前端加登入按鈕與愛心元件，以活動 id 讀寫 likes（見 §08）。
  8. 維護與監控來源改版時修對應 parser；Actions 失敗時 GitHub 自動寄信通知。
07 — Risks

需要先知道的風險

來源分散且無 API

每個來源各需一支 parser，網站改版就要維護。建議從少數穩定來源起步，不要一開始就追求全面。

Facebook 反爬與條款

FB 公開活動涵蓋最廣，但抓取困難且違反條款風險高。建議改以人工補登，或僅貼出連結。

「正面評價」的來源與真實性

感謝語多來自報名頁或社群留言，需註明出處或取得同意，避免變成不實宣傳。

資料正確性

爬來的日期、價格可能過期或誤判。卡片應標示「資料更新日」，並保留原始報名連結讓使用者核對。

08 — Community

登入與愛心評價

讓登入的人替活動按一顆愛心（可取消），網站顯示每場的愛心總數。重點：愛心是「會被很多人同時讀寫、且要長期保存」的資料，不能放進每天被爬蟲覆寫的 events.json，必須交給一個能驗證身分又能存資料的後端。

選用 Firebase（免費、不必自架伺服器）

前端維持 GitHub Pages 純靜態，只在頁面引入 Firebase JS SDK，由 Google 託管的服務承接登入與資料：

登入
Firebase Authentication 內建 Google 登入（OAuth），一行 signInWithPopup 即可，免處理密碼與 token。
→
資料
Cloud Firestore 存每場活動的愛心數與「誰按過」，前端即時讀取與更新。
→
規則
Security Rules 限定：未登入不可寫、每人對同一活動只能留一筆，防灌票。

資料怎麼存

愛心是動態層，與 events.json 用活動 id 對應，互不覆寫。建議用「一個使用者對一個活動一筆」的紀錄，總數即為筆數，天生防重複也能取消：

// Firestore：likes 集合，文件 id = `${eventId}__${uid}`
{
  "eventId": "2026-satir-taipei-0712",
  "uid":     "google-oauth 使用者唯一碼",
  "createdAt": "2026-06-23T10:20:00Z"
}
// 顯示某活動愛心數 = 查 likes where eventId == ? 的筆數
// 按愛心 = 新增該筆；取消 = 刪該筆（規則限 uid 必須等於登入者）

活動多時，可另存一份彙總計數（如 eventStats/{eventId}.likeCount）以 transaction 維護，前端就不必每次數整個集合。

取捨

  • 引入第三方託管後端（Firebase），原「完全無後端」改為「無自架後端」。Spark 免費方案的讀寫額度對這種規模綽綽有餘。
  • 需先建一個 Firebase 專案、開 Google 登入、設定授權網域（你的 Pages 網址）與 Firestore 安全規則。
  • 替代方案：Supabase（Postgres + Auth）功能相當；若不想綁 Google 生態可選它。建議二選一，不混用。
09 — Cost

成本

GitHub Pages 託管、GitHub Actions 排程（公開 repo 免費額度充足）、前端純靜態運算、Firebase 登入與愛心（Spark 免費方案）——全部 免費

唯一的「成本」是維護時間：來源網站改版時更新對應爬蟲；另需一次性建好 Firebase 專案與安全規則。
