結論（TL;DR）

基礎與技術型 SEO 已大致到位：SPA「原始 HTML 空白」這個最致命問題已解（活動清單 + ItemList<Event> 結構化資料皆於 build 時伺服器端預渲染進 index.html），robots / sitemap / GSC 驗證 / 意圖化 metadata 全部完成。

五份報告的最大共識：把「單頁」變「多頁」

5 份 AI 報告高度一致地指向同一個核心工程：在每日 build 時，由 build_site_json.py 遍歷 events.json，為每場活動／每位講師／每個地區靜態生成獨立 HTML 頁，並改用「一活動一頁」的單一 Event JSON-LD。這同時打破三道天花板：① 長尾關鍵字（特定講師／主題／地區）終於有專屬落地頁；② Event Rich Result 觸發率提高（Google 偏好一活動一頁，而非 ItemList 目錄頁）；③ 收錄頁數從 1 → 數十，站點規模與主題權威性同步上升。

次要但一致的共識：og:image 自動生成、過期活動分離、向來源機構爭取反向連結、GSC 提交 sitemap（這項最即時、零成本、純手動）。

本計畫所有「現況」欄位皆已對線上部署版 + 本機 repo 實測核對（非沿用舊簡報的宣稱值）。

報告之間的分歧點（值得注意）

  • SSG 該做多深 / 順序：Gemini／Manus 主張「絕對值得、最高優先」；Grok 較保守，建議「先從簡單版本起步」。本計畫採三階段分批上線：① 先為每場活動生成獨立頁 → ② 再為每位講師生成獨立頁 → ③ 最後為每個地區生成獨立頁。活動頁資料現成、零歧義，先取得長尾與 Event Rich Result 主效益；講師頁需先建同名／別名對照表；地區頁為聚合頁，放最後彙整。
  • URL 命名：各家不一（/events/{id} vs /events/{date}-{organizer}-{slug}、/teachers/ vs /instructors/ vs /organizers/）。本計畫對齊現有資料模型欄位（id／facilitator／region）統一命名，見 §3。
  • repo 膨脹：僅 Gemini 點出檔案數會隨活動增長，建議生成檔放 gh-pages 分支保持 main 乾淨。本計畫採納為實作備註。
  • 自訂網域：Gemini 直言這項違反「全免費」（約每年數百元台幣），Grok 則提免費 DNS 可能性。屬「破硬限制」項，列為選配。

4重點工程：靜態多頁生成（SSG）

這是 5 份報告的最大公約數，也是本架構唯一的長尾天花板。以下整合各家方案，落到本專案實況。

A. 三階段分批上線（依此順序實作）

本計畫把 SSG 切成三批，每批可獨立上線、各自見效，降低單次改動風險：

階段產出內容與理由
① 活動/events/{id}.html先做。每場有效活動一頁 + 單一 Event JSON-LD。資料現成、零歧義，直接拿下長尾與 Event Rich Result 主效益。
② 講師/facilitators/{slug}.html再做。彙整該講師所有活動（如「李崇建 薩提爾課程」）。前置需建同名／別名正規化對照表，故排第二。
③ 地區/regions/{region}.html最後做。彙整該地區活動（如「台中 薩提爾工作坊」），補齊地區型長尾的聚合頁。

B. 資料夾結構與 URL（對齊現有資料模型）

各報告命名不一，本計畫統一對齊 events.json 既有欄位（id／facilitator／region），避免自創欄名：

/
├── index.html              # 主清單頁（保留現有 SPA + 預渲染）
├── events/                 # ① 階段一：每場有效活動一頁
│   └── {id}.html           #    id 已存在且穩定，無 slug 碰撞
├── facilitators/           # ② 階段二：講師彙整頁（需同名/別名正規化）
│   └── {slug}.html
├── regions/                # ③ 階段三：地區彙整頁（台北／台中／高雄…）
│   └── {region}.html
├── archive/                # 過期活動歸檔（見 §6）
│   └── index.html
└── sitemap.xml             # 每階段把新 URL 併入

為何用 {id} 而非 {date}-{organizer}-{slug}：id 已是穩定主鍵，避免標題改字造成 URL 變動與 404；語意化交給頁內 <title> 與 h1 即可。

C. SPA 與多頁共存（Gemini 關鍵手法）

首頁卡片的 <a href> 寫死指向靜態活動頁（爬蟲爬到真實長尾 URL）；同時 JS 攔截點擊（e.preventDefault()）以 Modal 顯示詳情，保留 SPA 流暢體驗。爬蟲得到 URL，使用者得到無縫操作。

D. 每頁 SEO 元件

  • 獨立 title：「{活動名稱}｜{講師}・{地區}薩提爾工作坊 2026」
  • 獨立 JSON-LD：捨 ItemList，注入單一完整 Event（name／startDate／location／organizer／performer／offers），最大化 Rich Result。
  • canonical 指向自身；og:image 用該活動卡圖（§5）。

E. 實作落點（擴充 build_site_json.py）

現有 inject_index() 已用標記區注入首頁。新增 build_pages()，依階段順序實作：先 build_event_pages()（遍歷有效活動）→ 再 build_facilitator_pages()（依正規化後的講師分組）→ 最後 build_region_pages()（依地區分組），各自把 URL 併入 write_sitemap()。用 Python 字串模板（或 jinja2）輸出檔案。每日 Actions build 一併產生、納入 commit。77 場活動的生成量對 GitHub Pages 極輕量。

風險與維護（Gemini 提醒）

  • repo 膨脹：活動增加→檔案數增加。建議生成的 /events/ 等只存在 gh-pages 部署分支，保持 main 乾淨；或在 workflow 裡產生後直接部署、不回寫 source。
  • 過期 404：活動過期下架後，舊 URL 應 301/410 或導回 /archive/；至少別讓 sitemap 續列死連結（沿用現有 verify_links.py 精神）。
  • 講師正規化：同一講師多種寫法（全名／別名）會切碎權重，故講師頁列第二階段，先建對照表。

5og:image 自動生成（5/5 共識）

社群分享無縮圖直接壓低 CTR；圖片搜尋也零曝光。純靜態下可在 build 時自動產圖。

  • 原理：GitHub Actions 安裝 Pillow，準備一張品牌底圖，腳本把「活動名稱／日期／講師／地區」浮字壓上，輸出 /assets/og/{id}.png。
  • 實作：在 build_site_json.py（或新檔 build_og.py）加繪圖邏輯；各活動頁與首頁寫入 <meta property="og:image"> 與 <meta name="twitter:card" content="summary_large_image">。
  • 成本／風險：新增一個 Python 依賴 + 圖檔體積；CJK 字型需在 Actions 環境備妥（指定 Noto Sans TC 路徑）。
  • 效益原理：社群點擊回流＝正向使用者行為訊號，間接利於搜尋排名。

6過期活動分離（Gemini／Grok 高優先）

交易型意圖高度依賴「未來可報名」狀態。線上實測仍混入 2020–2021 活動，會稀釋頁面新鮮度與相關性。

  • 實作：build 時以資料更新日為界，把 events.json 切為 upcoming／past。首頁清單與預渲染、首頁 JSON-LD 只收 upcoming；過去活動進 /archive/（仍可索引，保留長尾與站點規模，但不佔首頁相關性）。
  • 順序：建議排在 SSG 之前——先界定「有效活動」，SSG 才只為它們生成頁，省工又避免大量過期頁。
  • 成本：低；純資料切分，無新依賴。

7站外權重與內容深度

反向連結冷啟動（5/5 共識，最高站外價值）

SatirDaily 本就是免費幫各機構曝光活動的公益平台，這是要連結的天然籌碼。備一封定稿信／FB 訊息，聯繫所爬取的 5 大來源：

  1. 說明：「我們做了一個公益薩提爾活動匯整站，每天自動幫貴單位活動增加曝光。」
  2. 請求：「方便的話，能否在貴站『相關資源／友站連結』提及 SatirDaily？」
  3. 只要 1～2 個高相關機構給 dofollow，對新站排名是質的跳躍。

內容深度（4/5）

  • 建 /guide/：「如何挑選薩提爾工作坊」「冰山理論入門」等知識文，內文自然連回相關活動／地區頁，建立站內主題權威。
  • 講師彙整頁附簡短 bio（須註明出處／取得同意，沿用專案「正評真實性」紀律）。

自訂網域（選配，破「全免費」）

綁定 .tw／.com 可脫離 github.io 子網域、累積獨立網域權重與品牌信任。Gemini 直言此項違反全免費硬限制（每年數百元），Grok 提免費 DNS 方案。屬高 SEO 上限、低急迫性的選配。
